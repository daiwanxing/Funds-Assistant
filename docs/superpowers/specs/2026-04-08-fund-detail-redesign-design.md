# 基金详情区重设计方案

日期：2026-04-08

## 背景

当前基金详情区采用顶部 `Hero` + 下方 `2×2` 网格布局。它能承载现有数据，但信息权重过于平均，导致净值走势没有成为主视觉，右侧关键信息也缺少“终端式”高密度扫描感。

这次重设计参考用户提供的截图，提炼其信息组织方式，而不是直接复刻视觉皮肤。目标是在保持现有数据源不变的前提下，把详情区改成“图表主导、右侧窄信息列、底部持仓明细始终可见”的结构。

## 目标

- 让净值走势成为详情区主角。
- 将基金概况、阶段收益、持仓摘要收敛为右侧窄信息列。
- 保留现有数据源与字段，不为这次改版新增接口依赖。
- 将完整持仓明细从四宫格中解放出来，改为底部始终可见的横向明细带。
- 维持当前深色终端风格，并与现有 dashboard 设计令牌保持一致。

## 非目标

- 不新增 AI 分析真实能力。
- 不改动左侧自选列表与顶部走马灯结构。
- 不新增基金详情额外远程字段。
- 不实现截图中的分钟级日内分时行情；图表仍基于现有累计净值走势数据。

## 布局方案

详情区改为三段式结构：

1. 顶部摘要条
2. 中部主分析区：左侧大图表 + 右侧窄信息列
3. 底部持仓明细带

### 1. 顶部摘要条

作用：建立当前选中基金的身份与核心行情感知。

展示内容：

- 基金代码
- 基金名称
- 最新累计净值
- 当前选中周期收益率
- 周期标签
- 少量元信息标签：基金类型占位文案、最低申购或费率等可用字段中的一个到两个

布局要求：

- 左侧显示基金身份信息。
- 右侧显示最新数值与涨跌语义。
- 视觉上弱化边框，增强数值主次关系。

### 2. 中部主分析区

中部区域使用左右分栏，比例建议约 `minmax(0, 1fr) 280px`。

#### 左侧：主图表区

展示内容：

- 时间周期切换按钮
- 当前周期区间标签
- ECharts 累计净值走势

设计要求：

- 图表成为视觉重心，卡片内边距略放大。
- 图表标题区与周期按钮保持在同一视觉带内。
- 保留现有涨跌红绿语义和面积渐变。
- 适当增加 Y 轴和 tooltip 的可读性，但不做数据含义变更。

#### 右侧：窄信息列

右侧按三个信息组垂直堆叠：

1. 基金概况
2. 阶段收益
3. 持仓概览

每组使用紧凑信息卡片，而不是松散的两列文本。

基金概况字段：

- 申购费率
- 原费率
- 最低申购

阶段收益字段：

- 近1月
- 近3月
- 近6月
- 近1年

持仓概览字段：

- 披露季度
- 前十大合计占比
- 第一大重仓股名称
- 披露持仓数量

说明：

- “前十大合计占比”由现有持仓接口返回列表求和得到。
- “第一大重仓股名称”取排序第一项。
- “披露持仓数量”即当前持仓列表长度。

### 3. 底部持仓明细带

完整持仓明细不进入右侧窄列，而是固定放在详情区底部，始终可见。

展示内容：

- 标题与披露季度
- 前若干条持仓明细
- 每条展示：排名、股票名称、市场标签、占比、横向进度条
- 底部总计占比

布局要求：

- 默认在单屏内直接可见，不依赖折叠。
- 与中部主分析区有清晰分隔。
- 保留纵向阅读节奏，但整体高度压缩，避免挤占图表主空间。

## 组件边界

### 保留组件

- `FundDetail.vue`
- `FundDetailHero.vue`
- `FundReturnChart.vue`
- `FundHoldings.vue`

### 新增组件

- `FundDetailSidebar.vue`
- `FundDetailInfoGroup.vue`

### 组件职责

`FundDetail.vue`

- 作为详情区编排层。
- 处理空、加载、错误、成功四种状态。
- 在成功态下拼装顶部摘要、中部左右结构、底部持仓明细带。

`FundDetailHero.vue`

- 只负责顶部摘要条。
- 接收基金身份、最新净值、周期收益与可选标签信息。

`FundReturnChart.vue`

- 只负责图表与周期切换。
- 保持当前 props / emit 方向，避免把概况信息重新耦合回图表组件。

`FundDetailSidebar.vue`

- 负责右侧窄信息列布局。
- 接收 `profile`、持仓摘要信息。
- 组合多个 `FundDetailInfoGroup`。

`FundDetailInfoGroup.vue`

- 负责单个信息组的标题、网格与数值样式。
- 作为纯展示组件，props 下发，零副作用。

`FundHoldings.vue`

- 从“四宫格面板”改为“底部明细带”。
- 继续自行消费 `useFundHoldings()`，避免父层承担持仓请求细节。

## 数据映射

不新增数据源，全部基于现有 composable：

- `useFundDetail()`：`profile`、`filteredTrend`、`periodReturn`、`periodLabel`、`latestNav`
- `useFundHoldings()`：`holdings`、`quarter`、`maxRatio`

需要新增的摘要计算：

- `topHoldingName`
- `holdingsCount`
- `holdingsTotalRatio`

这些摘要优先在 `FundHoldings.vue` 或独立的 sidebar 计算逻辑中通过 `computed` 派生，不引入新的 API 层。

## 状态与交互

- 空状态、加载态、错误态继续由 `FundDetail.vue` 统一控制。
- 周期切换仍由 `FundReturnChart.vue` 触发 `update:period`，父层调用 `setPeriod`。
- 右侧信息列不引入额外交互。
- 底部持仓明细带不折叠、不分页。

## 视觉原则

- 维持现有 `Bloomberg / terminal` 风格，不引入明亮卡片或大面积高饱和装饰。
- 通过布局主次，而不是大量颜色变化，建立视觉层级。
- 红涨绿跌语义保持不变。
- 卡片背景、描边、标题字重需与 `docs/design/dashboard-colors.md` 的 token 对齐。

## 验收标准

- 详情区不再使用 `2×2` 网格主结构。
- 图表在详情区中成为最大视觉区域。
- 右侧窄信息列明确分为“基金概况 / 阶段收益 / 持仓概览”三组。
- 持仓明细在底部始终可见。
- 不新增接口请求，现有测试和类型检查可通过。
- 在未选中基金、加载失败、数据为空时，布局退化仍然稳定。

## 实施顺序

1. 重构 `FundDetail.vue` 主布局。
2. 新增右侧信息列组件并迁移概况字段。
3. 调整 `FundDetailHero.vue` 的摘要排布。
4. 调整 `FundReturnChart.vue` 的头部与容器比例。
5. 重构 `FundHoldings.vue` 为底部明细带。
6. 运行 `test:run`、`type-check`、`lint` 验证。

## 风险

- 现有基金详情接口字段有限，右侧摘要区的信息密度需要通过排版而不是字段数量来补足。
- 底部持仓明细带若高度控制不好，会侵占图表空间，需要在视觉和可读性之间取平衡。
- `FundHoldings.vue` 目前同时承担数据与展示，若底部新样式过于复杂，后续可能需要继续拆分展示层。
